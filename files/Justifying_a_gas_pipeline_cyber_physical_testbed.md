# Justifying a gas pipeline cyber-physical testbed: datasets, physics, standards, and scenarios

A 20-node gas pipeline simulation built on MATLAB + CODESYS SoftPLC + Python Modbus/TCP gateway occupies a well-validated middle ground in the ICS security research fidelity hierarchy — above purely software-injected datasets like CIC Modbus 2023 and comparable in protocol authenticity to physical testbeds like SWaT and the Morris gas pipeline. This report synthesizes evidence across four interconnected domains to provide thesis-ready justification: (1) how the research community validates simulated ICS/SCADA datasets as genuine, (2) the scientific foundation and citation authority of GasLib-24 as a benchmark gas network, (3) how Indian CGD standards from BIS and PNGRB define realistic operating parameters, and (4) what operational scenario combinations the literature considers essential for diverse IDS training datasets. Together, these four pillars establish that a CODESYS-based gas pipeline testbed with physics-grounded parameters, protocol-authentic Modbus traffic, and comprehensive scenario coverage can produce a dataset that meets or exceeds the realism standards of leading ICS security benchmarks.

## How top papers define "near genuine" ICS datasets

The ICS security community evaluates dataset authenticity along a well-established fidelity hierarchy. Conti, Donadel, and Turrin (IEEE Communications Surveys & Tutorials, 2021) define three tiers: **physical testbeds** (real hardware and software, highest fidelity), **hybrid testbeds** (hardware-in-the-loop mixing real and virtual components), and **virtual testbeds** (lowest fidelity, entirely software-simulated). A CODESYS SoftPLC architecture falls squarely in the hybrid tier — it executes a genuine IEC 61131-3 runtime with authentic scan cycles while the physical process runs in MATLAB simulation.

Tantawy (De Montfort University, 2022, arXiv:2208.08255) proposes six essential criteria for CPS security datasets: time-stamped system inputs/outputs, measurable external disturbances, coverage of all normal operating modes plus probable failure modes, system model availability for augmentation, and — critically — **multi-modal data combining physical process and cyber/network layers**. Tantawy argues that "it may not be possible to distinguish between a cyber attack and a system failure by relying on physical system data only" and that cyber data fingerprints are essential for disambiguation. The ICS-Flow dataset paper (Dehlaghi-Ghadim et al., IEEE Access, 2023) adds three more criteria: comprehensiveness of ICS component representation, granularity of simulation, and reproducibility. These criteria collectively define the bar that a MATLAB + CODESYS + Modbus/TCP testbed must meet.

Seven specific realism markers recur across the top venues: **protocol authenticity** (real Modbus/TCP rather than simulated abstractions), temporal fidelity (realistic scan-cycle timing and jitter), physics-governed process dynamics, Purdue Model-compliant network topology, attack realism via actual tools aligned with MITRE ICS ATT&CK, dual-format data capture (PCAP network traffic and CSV process logs), and accurate ground-truth labelling with precise timestamps.

### Why a real SoftPLC matters more than Docker scripts

The distinction between CODESYS SoftPLC and purely script-based approaches like the CIC Modbus 2023 dataset (Boakye-Boateng, Ghorbani & Lashkari, PST 2023) is fundamental to dataset genuineness. CIC Modbus 2023 uses Docker containers where "Python scripts were generated to run the logic of IEDs and SCADA HMIs" and "the logic for an IED is to periodically change the voltage values randomly or when a request is received from SCADA HMI." No real PLC runtime executes a scan cycle; register changes are driven by Python event loops, producing none of the timing artefacts characteristic of real industrial controllers.

CODESYS, by contrast, is an **industrial-grade IEC 61131-3 runtime** deployed in hundreds of commercial PLC products from WAGO, Festo, Beckhoff, and others. It executes real Structured Text or Ladder Logic programs with configurable cyclic task times (10–100 ms), producing authentic scan-cycle jitter. Its native Modbus TCP server responds to function code requests (FC01–FC16) identically to hardware PLCs, with genuine 16-bit register quantisation where values stored as INT, UINT, or REAL undergo the same discretisation as physical PLCs. Alves (Computers & Security, 2018) noted in the OpenPLC paper that "creating a high-fidelity testbed for SCADA usually means having PLCs as hardware-in-the-loop, once it is really challenging to emulate a PLC in a virtual environment without information about the PLC internal behavior." SoftPLCs like CODESYS provide the same software execution environment at reduced cost, with the acknowledged trade-off being absence of hardware I/O timing.

The practical differences are measurable. Docker/Python approaches generate no scan-cycle periodicity, no register quantisation artefacts, and OS-level scheduling jitter that differs qualitatively from PLC task-scheduler jitter. A CODESYS setup produces inter-request intervals that follow a tight distribution around the configured scan time with **characteristic jitter of ~0.5 ms** (Găitan, MDPI Processes, 2025, measured 0.492 ms for Modbus RTU). Protocol fingerprinting research by Cook et al. (IFIP/IEEE IM 2019) showed that just three TCP/IP features — Time To Live, TCP window size, and vendor MAC ID — uniquely identify PLC devices, meaning a SoftPLC on Linux exhibits detectably different but internally consistent network signatures compared to Docker containers.

### Protocol-layer artefacts that prove real PLC communication

Four categories of Modbus/TCP protocol artefacts distinguish genuine PLC communication from synthetic injection. First, **scan-cycle periodicity**: real PLCs poll at fixed intervals (10–100 ms) with small jitter, producing sharp peaks in the power spectral density of inter-packet arrival times. Synthetic traffic tends toward either perfect periodicity (no jitter, physically unrealistic) or excessive randomness. Second, **register quantisation**: Modbus holding registers are 16-bit unsigned integers (0–65535), requiring physical values to be scaled from engineering units. Real PLC data shows discrete step changes corresponding to ADC resolution, while synthetic datasets often use floating-point values that lack this discretisation.

Third, **function code patterns**: real SCADA systems produce repetitive, deterministic function code sequences dominated by FC03 (Read Holding Registers), FC06 (Write Single Register), and FC16 (Write Multiple Registers). Modbus packets exhibit **characteristic uniform lengths concentrated at 66 and 85 bytes** (MDPI Symmetry, 2021). Transaction IDs increment sequentially, and Wireshark's `modbus.response_time` field captures authentic request-response pairing. Fourth, **inter-request timing distribution**: in real systems, the distribution of inter-request intervals is tightly clustered around the scan cycle with small variance; Găitan (2025) measured FC03 response times of 7.694–8.234 ms (mean 8.068 ms) for 10-register payloads, with server processing times of 4.978–5.837 ms. These timing signatures constitute a fingerprint that statistical tests can verify.

### Lessons from SWaT, Morris, and ICSSIM

The three most influential ICS benchmark datasets each justify genuineness differently — and each carries instructive limitations. **SWaT** (Mathur & Tippenhauer, SUTD, 2016) sets the gold standard: a 90-square-metre physical water treatment plant producing 5 gallons/minute of actual filtered water, designed in coordination with Singapore's Public Utility Board, using Allen-Bradley PLCs, EtherNet/IP, and real attack tools over 11 continuous days. Its justification rests on industrial collaboration, genuine hardware, and community validation through **300+ publications**. Yet even SWaT acknowledges "design limitations that make it difficult for system identification and attack detection in some experiments."

The **UAH Gas Pipeline dataset** (Morris & Gao, ICCIP 2014) used a laboratory-scale physical system with real Modbus communication, PID pressure control, and 28 real attack types. However, Morris himself later acknowledged a critical flaw: "some attacks were performed with gas pressure set to one value and normal operation with another," creating unintended patterns that gave ML classifiers artificially high accuracy. This cautionary example demonstrates that even real-hardware testbeds can produce unrealistic datasets if operational diversity is insufficient during attack injection — a lesson directly relevant to scenario design.

**ICSSIM** (Dehlaghi-Ghadim et al., Computers in Industry, 2023) takes the Docker container approach but justifies realism through OS-kernel isolation (each component runs in its own kernel generating real network traffic), actual Modbus TCP on port 502, physical process modelling, and MITRE ATT&CK-aligned attack scenarios. Notably, the authors explicitly acknowledge the need for "analysis to compare the network dataset collected in the simulated environment using container technology with the dataset collected in the physical testbed" — a validation step that a CODESYS SoftPLC setup inherently addresses by using an industrial-grade PLC runtime.

### Statistical validation of dataset realism

Five statistical tests form a standard battery for validating ICS dataset realism. The **Augmented Dickey-Fuller (ADF) test** verifies stationarity: steady-state process variables should be stationary (p < 0.05), while transients and setpoint changes should show non-stationarity. **Autocorrelation function (ACF) analysis** captures physical inertia — ICS process variables like tank levels and pipeline pressures exhibit strong lag-1 autocorrelation near 1.0 due to slow physical dynamics. The **SWaT dataset has obvious cycle patterns** in its autocorrelation structure, characteristic of real ICS data that simulated datasets must replicate.

The **Kolmogorov-Smirnov (KS) two-sample test** directly compares empirical cumulative distribution functions between simulated and reference datasets, applicable to inter-packet arrival times, Modbus response times, and sensor value distributions. KS Integral values below 80% indicate no statistically significant difference. **Power spectral density (PSD) analysis** reveals frequency content — real ICS data shows dominant frequencies from control loops, scan cycles, and process dynamics; PSD of Modbus polling intervals should show peaks at the scan-cycle frequency. The **Shapiro-Wilk test** verifies that steady-state sensor residuals follow Gaussian noise characteristics expected from physical measurement processes. For a CODESYS-based testbed, applying these five tests to both the process layer (MATLAB sensor outputs) and protocol layer (Modbus timing) provides rigorous dual-layer validation.

### Why dual-layer labelling elevates dataset credibility

Single-layer datasets — process-only or network-only — are the norm in ICS security research, but dual-layer (process + protocol) labelling provides three critical advantages. First, **cross-validation eliminates labelling errors**: when an attack label is applied, both layers should show correlated evidence — anomalous physics and anomalous Modbus traffic. The Morris gas pipeline dataset flaw, where fixed pressure values inadvertently signalled attacks, would have been caught if network-layer features independently confirmed attack presence. Second, dual-layer data enables **distinguishing attack vectors**: a false data injection attack shows sensor deviation at the process layer and anomalous write commands at the protocol layer, while a denial-of-service attack shows normal process dynamics but disrupted Modbus timing.

Third, dual-layer datasets support **multi-modal detection architectures**. Faramondi et al. (IEEE Access, 2021) explicitly designed their hardware-in-the-loop water distribution testbed to "encompass both physical and network data in order to highlight the consequences of attacks in the physical process as well as in network traffic behaviour." Tyab et al. (Journal of Cybersecurity, Oxford, 2021) demonstrated on the Morris gas pipeline dataset that multi-layer classification achieves **87.4% F-measure for malicious/benign detection, 74.5% for attack type, and 41.2% for specific attack variant** — showing that richer layer information enables progressively finer-grained detection. A MATLAB + CODESYS + Python gateway architecture naturally produces dual-layer data: MATLAB provides physics-based ground truth with process labels, while the Python Modbus/TCP gateway captures protocol-layer transactions with timestamps, function codes, register addresses, and response times.

## GasLib-24 rests on rigorous physics and German network data

The GasLib benchmark library, maintained at Zuse Institute Berlin (gaslib.zib.de), provides the most widely cited set of gas network test instances in the optimisation community. The core paper — Schmidt, Aßmann, Burlacu, Humpola, Joormann, Kanelakis, Koch, Oucherif, Pfetsch, Schewe, Schwarz, and Sirvent (Data, MDPI, 2017) — has accumulated approximately **138 citations** across venues including INFORMS Journal on Computing, Computers & Chemical Engineering, Applied Energy, Optimization Methods and Software, Mathematical Methods of Operations Research, and Nature Scientific Reports.

### The governing equations behind GasLib

GasLib's mathematical foundation is the **isothermal Euler equations**, a system of nonlinear hyperbolic PDEs governing compressible gas flow. As detailed in the companion monograph by Koch, Hiller, Pfetsch, and Schewe (Evaluating Gas Network Capacities, SIAM, 2015, 366 pp.), the gas dynamics follow mass conservation (∂ₜρ + ∂ₓ(ρv) = 0) and momentum conservation (∂ₜ(ρv) + ∂ₓ(p + ρv²) = −λ/(2d)·ρv|v| − ρg sin α), where λ is the **Darcy-Weisbach friction factor** computed via the Colebrook-White equation using each pipe's diameter, length, and inner wall roughness — all explicit parameters in GasLib data files.

The SIAM monograph presents a model hierarchy: **(M1)** full isothermal Euler equations, **(M2)** semilinear form neglecting the nonlinear convective term, and **(M3)** stationary algebraic equations of the Weymouth type. The **Weymouth equation** (1912), the oldest steady-state formulation, relates p₁² − p₂² to flow rate through pipe parameters (p₁² − p₂² = K·q|q|). The **Panhandle A equation** (1940s, for partially turbulent flow at Re 5–11 million) and **Panhandle B equation** (1956, for fully turbulent flow in large-diameter pipes at Re 4–40 million) provide alternatives with different friction-factor treatments. GasLib pipe elements carry diameter, length, and roughness parameters that enable any of these formulations, making the library agnostic to the specific steady-state equation chosen by researchers.

### Where GasLib's data comes from

GasLib instances derive from multiple real-world and synthetic sources. The larger instances (GasLib-582, GasLib-4197) are based on **confidential real-world data from Open Grid Europe GmbH** (formerly E.ON Gastransport), one of Germany's largest transmission system operators with ~11,000 km of pipeline. These data were "carefully distorted" to protect confidentiality while maintaining realistic hydraulic behaviour. **GasLib-134** closely approximates the real-world Greek gas transmission system (DESFA), with 1,412 km total length and 1,234 daily nominations sourced from the TSO's public website. The research was funded by the German Federal Ministry of Economic Affairs (BMWi) through the ForNe project (2009–2012) and by the DFG Transregio 154 on mathematical modelling of gas networks.

**GasLib-24 specifically** is described as "an extended variant of the network used in Ehrhardt & Steinbach (2005)" — an artificial but structurally representative small-scale test network with **3 entries, 5 exits, 16 inner nodes, 19 pipes, 3 compressor stations, 1 control valve, 1 short pipe, and 1 resistor**. GasLib-40 "roughly represents a part of the German low-calorific gas transport network in the Rhine-Main-Ruhr area" with 3 entries, 29 exits, 39 pipes, and 6 compressor stations. Both operate in the high-pressure transmission regime consistent with DIN EN 1594 (>16 bar).

### German standards underpinning GasLib parameters

Four key German/European standards define the engineering constraints reflected in GasLib parameters. **DIN EN 1594** governs gas infrastructure pipelines for maximum operating pressure over 16 bar — the foundational standard for high-pressure gas transmission in Germany. **DVGW G 463** applies specifically to high-pressure gas steel pipelines and works in conjunction with DIN EN 1594 for planning and construction. **EN 12186** specifies functional requirements for gas pressure regulating installations used at transmission-distribution interfaces — the standard underlying GasLib's control valve elements. **DVGW G 260** (revised September 2021 to incorporate hydrogen provisions) defines gas quality requirements including Wobbe index and contaminant limits for the German public gas supply, while **DVGW G 459** governs gas pressure regulation installations.

These standards explain GasLib's pressure ranges (typically **40–85 bar** for transmission networks), pipe materials (large-diameter steel per DIN EN 1594), and compressor/control valve configurations. The ACM e-Energy 2025 paper on Krylov solvers confirms that reference pressures for GasLib instances are set to the largest pressure found in the data, consistently in the transmission regime.

### A 20-node network is structurally consistent with GasLib

A 20-node gas pipeline network with 50 bar operating pressure, dual compressors, PRS stations, and underground storage aligns with GasLib-scale parameters across every dimension. In **node count**, GasLib-11, GasLib-24, and GasLib-40 bracket a 20-node network. In **pressure**, 50 bar falls squarely within the German high-pressure transmission range. In **compressor density**, GasLib instances show compressor-to-node ratios from 0.13 (GasLib-24: 3/24) to 0.21 (GasLib-135: 29/135); two compressors in 20 nodes (ratio 0.10) is conservative. GasLib-24 includes a control valve (functionally equivalent to a PRS), and GasLib-582 has 23 control valves. The inclusion of underground storage is a natural extension: while GasLib's current stationary instances don't explicitly model storage, Ríos-Mercado and Borraz-Sánchez (Applied Energy, 2015, 147:536–555) confirm that underground storage is a standard component of gas transmission systems, and the GasLib paper notes plans for transient extensions that would naturally include storage dynamics.

Numerous papers have validated against GasLib directly. Borraz-Sánchez et al. (INFORMS J. Computing, 2016) used GasLib-40, GasLib-135, and GasLib-582 for convex relaxations. Benner et al. (Springer DAE Forum, 2018) derived transient benchmarks from GasLib-134. The DAVE tool (Nature Scientific Reports, 2024) used GasLib-582 to calibrate roughness parameters for SciGRID gas data. Fokken et al. (Springer, 2021) combined IEEE power data with GasLib data for coupled gas-power simulation. This extensive validation ecosystem confirms GasLib as the de facto standard for gas network benchmarking.

## Indian CGD standards define a distinct pressure and topology regime

Indian City Gas Distribution networks operate under a regulatory framework jointly governed by the Bureau of Indian Standards (BIS), the Oil Industry Safety Directorate (OISD), and the Petroleum and Natural Gas Regulatory Board (PNGRB). These standards define operating parameters that differ substantially from German transmission networks — requiring specific parameter adaptations when using GasLib-24 as a structural template.

### The steel pipe standards hierarchy

Three BIS standards govern steel pipes in gas distribution. **IS 1239 (Part 1):2004** covers mild steel tubes from 6 mm to 150 mm NB in three pressure classes (light, medium, heavy), primarily used for above-ground galvanised iron service risers at domestic premises. **IS 3589:2001** specifies electrically welded steel pipes from 168.3 mm to 2540 mm OD in grades Fe 330, Fe 410, and Fe 510, rated up to 40 kg/cm², used for high-pressure steel trunk lines in CGD networks. **IS/ISO 3183:2007** (superseding the older IS 1978) provides the current line pipe specification aligned with API 5L. Additionally, **IS 14885:2001** specifies PE 80 (yellow) and PE 100 (orange) polyethylene pipes that form the backbone of medium- and low-pressure distribution networks — the most extensively used pipe material in Indian CGD by length.

**OISD-STD-141** (first issued April 1990, revised September 2003) governs cross-country hydrocarbon pipeline design based on ASME B 31.4, with OISD taking precedence in case of discrepancy. It defines location classes 1–4 with maximum inter-valve distances of 32, 24, 16, and 8 km respectively, wall thickness formula (tₙ = PD/2FESYMS), minimum 25-year design life, and mandatory cathodic protection. For CGD specifically, **OISD-STD-226** applies rather than STD-141.

### PNGRB regulations structure the pressure cascade

The PNGRB Technical Standards and Specifications (GSR 612(E), August 2008, with amendments through December 2024) constitute the master regulation for Indian CGD design. Based on ASME B 31.8 with Indian-specific modifications, they define three network tiers:

- **Primary network (steel)**: above 7 bar (100 psig) and below 49 bar (711 psig) — steel distribution mains and ring mains
- **Secondary network (MDPE)**: above 100 mbar (1.5 psig) and below 7 bar — medium-pressure distribution
- **Tertiary network (MDPE/GI)**: below 100 mbar — low-pressure service lines to domestic consumers

PE pipes are limited to PE 80 and PE 100 grades with a **maximum design pressure of 7 bar** and minimum wall thickness of 2.3 mm. Only electrofusion jointing is permitted for PE — butt fusion, solvent cement, and compression couplings are explicitly prohibited. Steel networks require hydrotesting at 1.4× MAOP.

### The Indian CGD pressure regime versus German transmission

The complete pressure cascade in an Indian CGD network differs fundamentally from German transmission:

| Network point | Indian CGD pressure | GasLib equivalent |
|---|---|---|
| Transmission pipeline inlet to CGS | 60–99 bar | Comparable to GasLib source pressures |
| City Gate Station outlet → steel grid | **26 bar** | Below typical GasLib 40–85 bar range |
| Steel pipeline grid | **14–26 bar** | Well below GasLib transmission regime |
| District Regulating Station outlet | **1–4 bar** | No GasLib equivalent (distribution level) |
| CNG mother station inlet | 14–26 bar (compressed to 250 bar) | Unique to Indian CGD |
| Domestic PNG delivery | **21 mbar** | No GasLib equivalent |

MGL (Mahanagar Gas Limited, Mumbai) operates its steel grid at **18 bar(g)** with 5 city gate stations serving 2.83 million households through 7,485 km of pipeline. IGL (Indraprastha Gas Limited, Delhi NCR) operates over 25,619 km of combined steel and MDPE pipeline serving 2.7 million domestic connections and 425 CNG stations. ATGL (Adani Total Gas Limited) covers 53 geographical areas across 125 districts with 27,011 inch-km of steel pipeline and 8,100+ km of MDPE.

### Adapting GasLib-24 parameters for Indian CGD

Translating a GasLib-24-inspired topology to Indian CGD conditions requires systematic parameter modification. **Source pressure** should reduce from GasLib's ~60–80 bar to **26 bar** (CGS outlet), with the steel grid operating at **14–26 bar**. Pipe diameters must shift from GasLib's large transmission sizes (DN 500–1000+) to CGD steel grid dimensions of **DN 100 (4″) to DN 300 (12″)**, with MDPE segments at OD 32–125 mm. Roughness values change substantially: steel pipe roughness remains ~0.045–0.05 mm, but MDPE pipes have roughness of only **0.003–0.007 mm** — an order of magnitude smoother.

The most significant structural change involves **replacing compressor stations with pressure regulating stations**. GasLib compressors boost pressure for long-distance transport; Indian CGD has no in-line compressors, only CNG compressors (14–26 bar to 250 bar). DRS units reduce pressure from 14–26 bar to 1–4 bar using active + monitor regulator pairs with slam-shut valves. PRS setpoints should be configured as: CGS at 60–99 → 26 bar, DRS at 14–26 → 1–4 bar, and service regulators at 1–4 → 0.11 bar. The network topology also shifts from GasLib's meshed/looped transmission structure to a predominantly **radial/tree structure** with limited ring mains in the steel grid — though the steel grid layer maintains some loop connectivity for reliability.

A notable **research gap** exists: no published work was found on cyber-physical security of Indian CGD networks specifically, and Paladugu, Korlepara, and Sanka (Materials Today: Proceedings, 2022) appears to be one of very few papers on Indian CGD simulation, using AVEVA PIPEPHASE. The adaptation of GasLib-24 to Indian CGD parameters represents a novel contribution to the literature.

## Operational scenarios that matter for pipeline IDS datasets

The literature identifies a systematic hierarchy of operational scenarios critical for gas pipeline reliability analysis and, specifically, for generating diverse IDS training datasets. The Morris gas pipeline dataset's well-documented flaw — where attacks were performed at one pressure and normal operation at another, enabling classifiers to learn operational state rather than attack signatures — makes the case that **each attack type must be injected across multiple operational baseline scenarios**.

### N-1 contingency and component criticality ranking

The EU gas infrastructure regulation mandates **N-1 analysis**: the system must satisfy total demand during a statistically once-in-20-years peak day even after losing the single largest infrastructure element. Clees (Mathematical Methods of Operations Research, 2022) extends this through a **supernode** approach where entire compressor stations, regulator stations, and supply facilities are treated as single contingency elements — more realistic than individual component analysis. The paper's adapted N-1 and multi-element contingency method applies stationary Euler equations with advanced compressor thermodynamics. Schröder et al. (Applied Mathematical Modelling, 2024) develop a "highly robust" simulation approach specifically designed for states far from the design point — critical for contingency analysis where the system operates in unusual configurations.

Components rank by criticality as follows: **source/supply nodes** (loss causes complete supply disruption), **compressor stations** (loss reduces throughput capacity by ~250 psig per station according to NETL, 2017), key pipeline segments (loss isolates network sections), **pressure regulation stations** (loss affects downstream delivery pressure), and critical valves (failure prevents isolation or reconfiguration). Gas networks differ from electric power systems in that **linepack provides a time buffer** before pressure violations occur, enabling "survival time" analysis using tools like SIMONE.

### Source, demand, and compressor scenario design

Source failure scenarios range from complete outage to partial pressure reduction. Chen et al. (Reliability Engineering & System Safety, 2021) model supply reliability using Markov stochastic processes, finding that **cross-over valves play a critical role** in supply reliability under pipe segment failures. The real-world Winter Storm Uri (February 2021) demonstrated the consequences of a 20% reduction in wholesale gas supplies simultaneous with record heating demand, forcing curtailment to 28,000 New Mexico customers.

Demand variation follows well-characterized patterns: **1-in-20 peak day** (UK standard) or 1-in-10 (US standard) for capacity planning, diurnal cycling with peaks at 6–8 AM and 5–7 PM, and seasonal winter/summer variation. Chen et al. (Reliability Engineering & System Safety, 2021) propose adequacy, availability, and shortage rate indices under transient hydraulic-thermal simulation. Stochastic demand modelling typically uses Latin Hypercube Sampling with Cholesky decomposition, with Imperial College (2021) using 1,000 scenarios reduced to 5 representative scenarios for tractability.

The JRC Report (2020, JRC120149) provides the definitive taxonomy of **compressor station failure modes**, defining damage classes as percentage of nominal outlet flow available post-failure. Over 160 compressor failure scenarios were considered in Reliability Engineering & System Safety (2022) for optimising standby schemes. Key scenarios include: both units running (normal), single unit running (partial redundancy), compressor bypass (flow continues at reduced pressure), and compressor trip/surge (emergency shutdown). Bypass operation results in approximately 250 psig pressure loss per station.

### Valve, PRS, and storage scenarios round out the operating envelope

Valve failure modes follow three paradigms: **fail open** (dangerous when isolation is needed), **fail closed** (dangerous when continued flow is needed), and **fail last position** (when either change is unacceptable). Partial valve opening creates a throttling restriction modellable as increased resistance. The UK HSE documented a real case where a spring-return ESDV's mechanism corroded, leaving the primary platform isolation device stuck open. Closing isolation valves fundamentally changes network topology by creating disconnected subgraphs; opening storage loop valves (E14/E15) creates additional loops that provide alternative flow paths, change pressure distribution, and enable storage injection/withdrawal.

Leo et al. (Computers & Chemical Engineering, 2020) provide the most rigorous PRS failure modelling, studying four fault types through first-principles simulation: **spring decay** (slow outlet pressure drift), **filter choking** (reduced inlet pressure from particle accumulation), **diaphragm malfunction** (cyclic fatigue with ~200,000 cycle lifetime), and **valve seat damage** (rapid over-pressure or flow stoppage). The real-world Merrimack Valley incident, caused by a PRS over-pressure event, led to the US PIPES Act 2020 mandating operator assessments of district regulator stations.

Storage operations add two fundamental modes: **injection** (summer low-demand storage filling) and **withdrawal** (winter high-demand gas extraction). Linepack serves as a short-term virtual storage mechanism — packing increases pipeline pressure by injecting more than withdrawing, while drafting depletes stored gas down to a minimum of typically 60% of operating pressure for API-5L steel. Research in Journal of Modern Power Systems (2019) demonstrated that linepack optimisation across a network can save significant operating costs by time-shifting gas purchases.

### The complete scenario matrix for IDS dataset generation

Synthesising the literature, the priority scenario hierarchy for a 20-node gas pipeline IDS dataset is:

1. **Normal baselines at multiple operating points** — steady-state at peak, off-peak, and intermediate demand levels (prevents the Morris dataset flaw)
2. **Source variations** — single source outage, partial pressure reduction, dual-source operation
3. **Compressor configurations** — both running, single running, bypass, trip event
4. **Demand cycling** — diurnal peak/off-peak patterns, seasonal variation, stochastic fluctuations
5. **Valve operations** — isolation valve states, storage loop (E14/E15) open versus closed (topology change)
6. **PRS scenarios** — normal regulation, setpoint drift, PRS offline/bypass
7. **Storage modes** — injection, withdrawal, idle, linepack packing/drafting
8. **N-1 contingencies** — systematic single-component failures
9. **Combined scenarios** — source failure + peak demand, compressor trip + stuck valve
10. **Cyber attack overlays across all modes** — command injection, false data injection, DoS applied across multiple operational baselines

The literature on gas network resilience classifies operational modes along four dimensions (MDPI Energies Review, 2023): **deterministic disturbances** (known failures), **probabilistic disturbances** (random failures), **supply-demand imbalances**, and **information network disturbances** (SCADA/cyber attacks). Chen et al. (Reliability Engineering & System Safety, 2024) integrate Bow-Tie Analysis with Dynamic Bayesian Networks, demonstrating that initial failure probability decreases from 0.03767% to 0.01435% when connected to a resilience network — quantifying the value of redundancy in scenario-diverse datasets.

## Conclusion: four pillars of testbed justification converge

This research establishes that a MATLAB + CODESYS SoftPLC + Python Modbus/TCP gateway for a 20-node gas pipeline simulation occupies a defensible and well-justified position in the ICS security research landscape. **Dataset genuineness** is supported by CODESYS's industrial-grade IEC 61131-3 runtime producing authentic scan-cycle jitter, register quantisation, and function-code patterns verifiable through ADF, KS, ACF, PSD, and Shapiro-Wilk statistical tests — surpassing Docker-based approaches like CIC Modbus 2023 and addressing the dual-layer labelling gap that weakened the Morris gas pipeline dataset. **Physical validity** derives from GasLib's 138-citation benchmark library grounded in isothermal Euler equations, Darcy-Weisbach friction, and real German network data from Open Grid Europe, with a 20-node topology consistent with GasLib-24's scale, pressure regime, and compressor density. **Indian CGD alignment** — a novel contribution filling an identified research gap — is achievable through systematic parameter reduction from GasLib transmission pressures to the 14–26 bar steel grid and 1–4 bar MDPE distribution regime defined by PNGRB T4S regulations, with compressor stations replaced by DRS/PRS elements matching OISD and BIS standards. **Scenario diversity** addresses the critical lesson from the Morris dataset by overlaying cyber attacks across a minimum of 10 operational modes spanning source, demand, compressor, valve, PRS, and storage configurations — consistent with the N-1 contingency analysis mandated by EU gas regulation and the multi-dimensional resilience classification frameworks in the 2022–2025 literature.

The convergence of these four dimensions — protocol authenticity, physics fidelity, standards compliance, and operational diversity — provides a thesis-ready justification framework where each pillar reinforces the others. The dual-layer labelling inherent in the architecture (MATLAB physics ground truth + Python Modbus/TCP protocol capture) enables the cross-validated, multi-modal datasets that Tantawy (2022), Faramondi et al. (2021), and the broader ICS security community identify as the next standard for credible CPS security research.
