## 1. The 48-Hour Continuous Sweep

### Why `run_24h_sweep` Is Wrong for Realistic Data

Your current sweep treats each scenario as an **isolated universe** — fresh state, fresh EKF, fresh PID integrators. A real pipeline never does this. The state at hour 12 depends on everything that happened in hours 0–11.

**`run_48h_continuous.m`** (created above) fixes this by:

| Feature | run_24h_sweep (old) | run_48h_continuous (new) |
|---|---|---|
| State continuity | Reset every scenario | Single continuous 48h |
| Demand pattern | Flat per scenario | Realistic diurnal (2 day/night cycles) |
| Regime transitions | Instantaneous + state reset | Smooth cfg change, no state reset |
| Attack placement | Per-scenario, restarted | Random across entire 48h window |
| PID integrator | Reset each scenario | Carries wind-up across regimes |
| EKF covariance | Reset to P0 each time | Evolves continuously |
| Storage inventory | Fixed init per scenario | Drifts with inject/withdraw cycles |

### Regime Timeline (What Happens Over 48h)

```
Hour 0-6   : Night — low demand, both sources, storage injecting
Hour 6-10  : Morning ramp — demand rising, compressors ramping up
Hour 10-14 : Daytime peak — full demand, storage withdrawing
Hour 14-16 : MAINTENANCE WINDOW — E8 forced open, CS2 offline
Hour 16-18 : Recovery — auto control restored
Hour 18-22 : Evening peak — cooking gas demand, high flow
Hour 22-24 : Night again — demand falling

Hour 24-30 : Night/morning repeat
Hour 30-34 : Morning peak
Hour 34-38 : S2 SUPPLY INTERRUPTION — single-source operation
Hour 38-42 : S2 restored, recovery
Hour 42-46 : Evening peak
Hour 46-48 : Night wind-down
```

Attacks are scattered randomly throughout (any attack during any regime).

### Usage

```matlab
% Full 48h dataset (~172,800 rows at 1 Hz)
run_48h_continuous()

% 24h version
run_48h_continuous('duration_h', 24)

% Heavy attack scenario for IDS stress-testing
run_48h_continuous('attack_density', 'high')

% With CODESYS gateway (adds protocol artefacts)
run_48h_continuous('gateway', true)
```

### Output

```
automated_dataset/continuous_48h/
  physics_dataset.csv      — 172,800 rows × ~130 cols (main ML input)
  operating_log.csv        — regime change timestamps
  run_metadata.json        — config dump for reproducibility
```

## 2. Files Needed for the ML Pipeline

The ML pipeline needs **both physics and protocol context**. Here's what to feed it:

### Primary: Physics Dataset

**File:** `automated_dataset/continuous_48h/physics_dataset.csv`

This is what `cgd_ids_pipeline.py` consumes. It contains:

| Column Group | Count | What It Gives ML |
|---|---|---|
| Pressures (p_*_bar) | 20 | Physical state — anomaly baseline |
| Flows (q_*_kgs) | 20 | Mass balance features |
| Equipment (CS1/CS2/PRS/STO) | 10 | Actuator state |
| CUSUM + chi2 detectors | 5 | Built-in anomaly scores |
| EKF residuals (ekf_resid_*) | 20 | Physics-model deviation |
| PLC registers (plc_p_*, plc_q_*) | 40 | What the SCADA system actually sees |
| Labels (ATTACK_ID, FAULT_ID, label) | 4 | Ground truth |
| regime_id | 1 | Operating context |

**Key insight:** The PLC columns (plc_p_*, plc_q_*) differ from physics columns (p_*_bar, q_*_kgs) because of zone-based polling staleness, sensor noise, and spoofing. The gap between `p_S1_bar` and `plc_p_S1` is itself a feature.

### Secondary: Protocol Dataset (Gateway Mode Only)

**File:** `middleware/logs/modbus_transactions_*.csv`

Generated by `gateway.py` when running with `'gateway', true`. Contains:

| Column | What It Gives ML |
|---|---|
| timestamp_ms | Inter-packet timing (jitter fingerprinting) |
| fc (FC3/FC16/FC1) | Modbus function code distribution |
| direction (READ/WRITE) | Traffic pattern |
| modbus_addr | Register access pattern |
| int16_raw | Raw integer before engineering conversion |
| eng_value | Scaled engineering value |

**File:** `middleware/logs/pipeline_data_*.csv` (150 columns)

Generated by `data_logger.py`. Contains both engineering values AND raw integers for all 70 registers + 7 coils simultaneously. This is what a real network IDS would see.

### What to Feed the Pipeline

```bash
# Physics-only (sufficient for most IDS research):
python cgd_ids_pipeline.py --data ../automated_dataset/continuous_48h/physics_dataset.csv

# Physics + protocol (full CPS context):
# Merge the two CSVs on timestamp alignment first, then:
python cgd_ids_pipeline.py --data merged_physics_protocol.csv
```

### Merging Physics + Protocol

To get both in one file, you need a merge script:

```python
# merge_datasets.py
import pandas as pd

physics = pd.read_csv('automated_dataset/continuous_48h/physics_dataset.csv')
protocol = pd.read_csv('middleware/logs/pipeline_data_latest.csv')

# Align on nearest timestamp (physics is 1 Hz, protocol is 10 Hz)
protocol['Timestamp_s'] = protocol['timestamp_ms'] / 1000.0
protocol_1hz = protocol.groupby(protocol['Timestamp_s'].round(0)).first().reset_index()

merged = physics.merge(protocol_1hz, on='Timestamp_s', how='left', suffixes=('', '_proto'))
merged.to_csv('automated_dataset/continuous_48h/merged_dataset.csv', index=False)
```

This gives you ~300 columns: physics state + raw Modbus integers + coil states.

## 3. Modifications for a General-Purpose Digital Twin

Your simulator is currently hardcoded to a specific 20-node Indian CGD topology. To make it a **configurable digital twin** that adapts to any architecture, you need these changes:

### A. Topology Must Be Data-Driven (Not Hardcoded)

**Current problem:** `simConfig.m` has `cfg.edges`, `cfg.nodeNames`, `cfg.nodeTypes` all hardcoded for 20 nodes and 20 edges. Every function assumes `nNodes=20`, `nEdges=20`.

**Fix:** Load topology from a JSON/YAML file:

```json
// topology.json
{
  "nodes": [
    {"id": 1, "name": "S1", "type": "source", "elevation_m": 0, "volume_m3": 100},
    {"id": 2, "name": "J1", "type": "junction", "elevation_m": 5, "volume_m3": 80},
    ...
  ],
  "edges": [
    {"id": 1, "from": 1, "to": 2, "D_m": 0.2032, "L_km": 5.0, "type": "pipe"},
    {"id": 8, "from": 2, "to": 9, "D_m": 0.1524, "L_km": 6.5, "type": "valve"},
    ...
  ],
  "compressors": [
    {"node": 3, "ratio_min": 1.1, "ratio_max": 1.6, "head_curve": [50000, -100, -0.5]}
  ],
  "prs_stations": [
    {"node": 10, "setpoint_bar": 18.0, "deadband_bar": 0.5, "tau_s": 30.0}
  ]
}
```

**Files to modify:**
- `simConfig.m` → `loadTopology(json_path)` that builds cfg from file
- `initNetwork.m` → Remove hardcoded 20-node assumptions
- `initCompressor.m` → Support N compressors (not just 2)
- `initPRS.m` → Support N PRS stations
- `updatePLC.m` → Zone assignments derived from topology, not hardcoded arrays

### B. EKF State Dimension Must Be Dynamic

**Current:** `ekf.nx = 40` hardcoded (20p + 20q).

**Fix:** `ekf.nx = nNodes + nEdges` — already partially done but several functions still assume 20/20 splits. Audit:
- `updateEKF.m` — uses `nN = numel(meas_p)` ✓ (already dynamic)
- `buildJacobian` — uses hardcoded `nN=20` in some paths
- `computeFDIVector.m` — `nN = 20; nE = 20;` hardcoded ✗
- `initCUSUM.m` — `cusum.nx = 40` hardcoded ✗

### C. Attack Catalogue Must Be Pluggable

**Current:** `applyAttackEffects.m` has a `switch aid` with cases 1-8 hardcoded.

**Fix:** Attack registry pattern:

```matlab
% attacks/attackRegistry.m
function registry = attackRegistry()
    registry = containers.Map();
    registry('A1_SrcPressure')  = @attack_src_pressure;
    registry('A2_CompRatio')    = @attack_comp_ratio;
    % User adds custom attacks here
    registry('A11_CustomValve') = @attack_custom_valve;
end
```

Then `applyAttackEffects` becomes:
```matlab
fn = registry(attack_name);
[outputs...] = fn(inputs...);
```

### D. Real-Time Data Ingestion Interface

**Current:** The gateway only talks to CODESYS via Modbus.

**For a true digital twin** you need:
1. **OPC-UA client** — the industry standard for real SCADA/DCS communication
2. **MQTT subscriber** — for IIoT sensor feeds
3. **Historian reader** — pull from OSIsoft PI / Wonderware InSQL
4. **REST API** — for cloud-hosted twin

**Add:** `interfaces/` folder with:
```
interfaces/
  opcua_client.py      — read/write via OPC-UA (python-opcua)
  mqtt_subscriber.py   — subscribe to sensor topics
  historian_reader.py  — query PI/Wonderware for calibration data
  rest_api.py          — Flask/FastAPI wrapper around gateway.py
```

### E. Model Calibration Loop

A digital twin isn't just a simulator — it **adjusts its parameters to match the real plant**.

**Add:** `calibration/` folder:
```
calibration/
  calibrate_pipe_params.m    — fit pipe_rough, pipe_D from measured ΔP vs Q
  calibrate_compressor.m     — fit head curve coefficients from operating data
  calibrate_ekf_noise.m      — tune Q/R from innovation statistics
  calibrate_demand_profile.m — fit diurnal demand from billing/meter data
  auto_calibrate.m           — orchestrator: pull historian data → fit → update cfg
```

The calibration loop:
```
Real plant data (via OPC-UA/historian)
        ↓
Compare: measured vs simulated (residuals)
        ↓
Optimise: cfg.pipe_rough, cfg.comp1_a1, etc.
        ↓
Update: simConfig parameters
        ↓
Re-run: verify residuals decreased
```

**P0 is the most urgent** — the physics divergence means your pressures hit 0.1/70 bar clamps, which isn't realistic. Increase `cfg.node_V` from 100 to 500-1000 m³ and add a relaxation factor (0.3-0.5) in `updatePressure.m`:

```matlab
% In updatePressure.m, change:
p = p + coeff .* (params.B * q);
% To:
relax = 0.3;  % under-relaxation for stability
p = p + relax * coeff .* (params.B * q);
```

This single change will likely eliminate the oscillation divergence and make all downstream data realistic.
